# ADR 0001: 値オブジェクトを活用した、バリデーションの仕方

- **ステータス**: 採用
- **作成日**: 2026-05-29

## コンテキスト

これまでは値オブジェクトの検証に例外を採用しており、 Command を組み立てる際に一つでも不正なフィールドがあると即座に例外が投げられていた。

```kotlin
val command = UpdateUserCommand(
    id = userId,
    name = Name(request.name),                      // ← ここで throw すると API の利用者は Name の検証エラーにしか気づけない
    phoneNumber = PhoneNumber(request.phoneNumber), // ← 次のフィールドのエラーに気づけない
    postalCode = PostalCode(request.postalCode),
    address1 = Address1(request.address1),
    address2 = Address2(request.address2),
)
```

この方式だと、 name と phoneNumber の両方が不正でも **最初に評価された name の例外しかクライアントに返せない**。  
クライアントは「name を直して再送 → 今度は phoneNumber で弾かれる」 という往復を強いられる。

<br>

## 決定

例外ベースの「最初の 1 件で即 throw」 をやめ、 **値オブジェクトの検証結果を `Result` 型で表現し、 複数フィールドのエラーを蓄積してから一括で返す** 方式に切り替える。

具体的には次の 3 点を採用する。

1. **値オブジェクトは `create()` ファクトリで `Result<T, ValidationError>` を返す**  
   constructor を直接公開せず、 検証を通った値だけがインスタンス化される。 不正な入力は例外ではなく `Err(ValidationError)` として返るため、 「検証に失敗しうる」 という事実が戻り値の型に現れ、 呼び出し側は握り潰せない。

2. **検証エラーは値オブジェクト内に `object` として列挙する**  
   `Name.Blank` / `Name.TooLong` のようにエラーを値オブジェクトの名前空間へ閉じ込め、 検証ルールとクライアント向けメッセージを 1 ファイルに凝集させる。 ルール変更が 1 箇所で完結する。

3. **複数フィールドの検証は `zipOrAccumulate` で蓄積する**  
   1 件目で打ち切らず全フィールドを評価し、 発生した `ValidationError` を `List` にためる。 全件成功したときだけ Command が組み上がり、 1 件でも失敗すれば溜まったエラーを `ValidationException` にまとめて gRPC 層へ送る。

この結果、 クライアントは name と phoneNumber が同時に不正でも **1 回のレスポンスで全エラーを受け取れ**、 CommandHandler に届く時点では **全フィールドが検証済みである保証が型レベルで得られる**。 各ステップの実装は以下のとおり。

### 1. エラー構造と例外

```kotlin
abstract class ValidationError(
    val field: String,
    val message: String,
)

class ValidationException(
    val errors: List<ValidationError>,
) : RuntimeException(
        errors.joinToString(separator = " / ") { "[${it.field}] ${it.message}" },
    )
```

### 2. 値オブジェクト

```kotlin
// 1. Axon の Command に含める都合により、 constructor は internal とする
data class Name internal constructor(val value: String) {
    companion object {
        const val MAX_LENGTH = 100
        
        // 2. ファクトリメソッドで、 ValidationError を内包した Result 型を返却する
        fun create(input: String): Result<Name, ValidationError> {
            if (input.isBlank()) return Err(Blank)
            if (input.length > MAX_LENGTH) return Err(TooLong)
            return Ok(Name(input))
        }

        // 3. エラーは object 型として値オブジェクト内に列挙する
        object Blank : ValidationError("name", "名前は必須です")
        object TooLong : ValidationError("name", "名前は $MAX_LENGTH 文字以内で入力してください")
    }
}
```

### 3. バリデーション

```kotlin
        val commandResult =
            // 1. zipOrAccumulate 関数は発生したエラーを蓄積して List 形式で返却する
            zipOrAccumulate(
                { Name.create(request.name) },
                { PhoneNumber.create(request.phoneNumber) },
                { PostalCode.create(request.postalCode) },
                { Address1.create(request.address1) },
                { Address2.create(request.address2) },
            ) { name, phoneNumber, postalCode, address1, address2 ->
                UpdateUserCommand(
                    id = userId,
                    name = name,
                    phoneNumber = phoneNumber,
                    postalCode = postalCode,
                    address1 = address1,
                    address2 = address2,
                )
            }

        commandResult
            // 2. 失敗だったらそのまま例外にラップし、 gRPC のインターセプターにキャッチしてもらう
            .onFailure { errors -> throw ValidationException(errors) }
            // 3. 成功ならそのまま Command を送信する
            .onSuccess { command -> commandGateway.updateUser(command).throwIfError() }
```

### 4. 想定される gRPC エラーレスポンス

`ValidationException` は gRPC 層で `INVALID_ARGUMENT` (code `3`) にマッピングされ、 蓄積した field 別エラーは `google.rpc.Status.details` の `ErrorDetail.validationError` に乗る (変換の詳細は [ADR 0002](./0002-grpc-error-response.md))。

name と phoneNumber が同時に不正だった場合、 クライアントが受け取る `Status` の JSON 表現は次のようになる:

```json
{
  "code": 3,
  "message": "[name] 名前は必須です / [phoneNumber] 電話番号は ハイフン区切り (例: 090-0000-0000) で入力してください",
  "details": [
    {
      "@type": "type.googleapis.com/momiji.common.v1.ErrorDetail",
      "validationError": {
        "errors": [
          { "fieldName": "name", "message": "名前は必須です" },
          { "fieldName": "phoneNumber", "message": "電話番号は ハイフン区切り (例: 090-0000-0000) で入力してください" }
        ]
      }
    }
  ]
}
```

## 妥協点

### 1. なぜ値オブジェクトのプライマリコンストラクタにおいて、 `private constructor` ではなく `internal constructor` を使っているか

Axon が Command を AxonServer に送る際に **Jackson でシリアライズ** する。 Jackson の reflection は JVM 可視性を要求するため `private constructor` だと `Cannot construct instance` でデシリアライズが失敗する。

- Kotlin `private`  → JVM `private` → Jackson シリアライズ不可
- Kotlin `internal` → JVM `public`  → Jackson シリアライズOK

完全 `private` を維持したい場合は Custom Jackson Deserializer を 1 個ずつ書く必要があり、 コストが見合わない。 `internal` + 後述の `-Xconsistent-data-class-copy-visibility` で「外部モジュールからのcopyメソッドが使用されるのを禁止」 できるので実用的な妥協点とする。

### 2. なぜ `-Xconsistent-data-class-copy-visibility` を追加したか

Kotlin の `data class` は primary constructor が `internal` でも、 自動生成される `copy()` がデフォルトで `public`となる。 つまり  

```kotlin
val n = Name.create("Alice").get()!!
val sneaky = n.copy(value = "")  // ← バリデーションを突破して不正な値を入れることができる
```

これを防ぐため compiler flag でこの設定を追加し、「copy() の可視性を primary constructor に揃える」。  
※将来 Kotlin の default 挙動になる予定らしいのでこの記述自体不要になるかも

<br>

## 帰結

### 良かった点:

- CommandHandler に渡る時点で全フィールドが validation 済みである保証が型レベルで得られる
- フィールド別エラーが全部集約されて 1 レスポンスで返るのでクライアントから見たUXは明らかに向上している
- 凝集度の高いエラー集約構造 (Name のルール変更は Name.kt 1 ファイル) で保守性が上がる

### 注意すべき点:

- `internal constructor` の妥協点を理解せず `private` にすると Axon serde で落ちる
