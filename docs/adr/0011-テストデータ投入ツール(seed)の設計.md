# ADR 0011: テストデータ投入ツール（seed）を gRPC command 経由 + 独立ソースセットで作る

- **ステータス**: 採用 ( `src/seed` ソースセット + `gradlew seedData` を導入。 ローカルで実行・read model 反映を確認済み )
- **作成日**: 2026-06-06
- **関連 ADR**: [0010 ハードデリートを廃しライフサイクルを採用する](./0010-ハードデリートを廃しライフサイクルを採用する.md)（seed が作る brand/product のモデル） / [0002 エラー分類と gRPC での取り扱い](./0002-エラーの種類と分類方法、gRPC層での取り扱いについて.md)（azp 検証の文脈）

## コンテキスト

ローカル開発・admin 画面の確認のために、 brand / product のテストデータをまとめて投入したい。 momiji は CQRS + Event Sourcing（Axon 5.1.1 DCB）なので、 「どう投入するか」がツールの形を決める。

### 経緯 ( なぜ ADR にするか )

- **read DB に直接 INSERT してはいけない。** read model はイベントの投影で、 直接 INSERT するとイベントストアと不整合になり、 リプレイ（再構築）で消える（[ADR 0008](./0008-プロジェクションの日時はイベントのTimestampから取得する.md) / [0010](./0010-ハードデリートを廃しライフサイクルを採用する.md) の前提）。 必ず **command を撃って events → projection** で作る必要がある。
- command を撃つ手段は 2 つ: **CommandGateway を直接（アプリ内）** か **gRPC API（外部プロセス）**。 だが gRPC は認証必須で、 さらに `JwtClientIdValidator` が **token の azp == `momiji.oidc.client-id`（= `momiji`）** を検証する。 別クライアントのトークンは弾かれる。
- 「main とは別の tool 領域に置きたい」「一発ツールとして起動中の backend に撃ちたい」という要件がある。

これらの判断（どこに置くか・どう認証するか・冪等性）を明文化する。

## 決定

### 1. seed は gRPC command 経由で投入する（read DB 直 INSERT しない）

起動中の backend に `CreateBrand` / `CreateProduct` / `ArchiveBrand` / `DiscontinueProduct` を gRPC で撃つ。 events → projection の正規ルートを通るので read model と event store が常に整合する。

### 2. main とは別の独立ソースセット `src/seed` + `gradlew seedData`

`build.gradle.kts` で `seed` ソースセットを作り、 `JavaExec` task `seedData` で起動する。 main の output（生成済み gRPC stub 等）と runtime 依存（grpc / jackson 等）を借りるが、 **app jar には含まれない**。

```kotlin
sourceSets {
    create("seed") {
        compileClasspath += sourceSets.main.get().output + configurations.runtimeClasspath.get()
        runtimeClasspath += sourceSets.main.get().output + configurations.runtimeClasspath.get()
    }
}
tasks.register<JavaExec>("seedData") {
    mainClass.set("jp.momiji.seed.SeederKt")
    classpath = sourceSets["seed"].runtimeClasspath
    dependsOn("classes")
}
```

### 3. 認証は `momiji` クライアントの client_credentials

seeder は別クライアントでなく **`momiji` クライアント自身で `client_credentials`** トークンを取る。 こうすると token の **azp が `momiji`** になり `JwtClientIdValidator` を通る。 そのため realm の `momiji` クライアントに **`serviceAccountsEnabled: true`** を付与する（[momiji-realm.json](../../local/keycloak/momiji-realm.json) に追加。 起動中の Keycloak へは Admin REST の partial 反映でも可）。

### 4. 固定の決定的 ULID で冪等化

seed する id は `01JSEEDBRAND…` のような**固定の決定的 ULID**（26 文字・Crockford base32 の有効文字のみ）。 create は id がキーで冪等なので、 **何度 `seedData` を流しても重複しない**（再実行安全）。 規模は brand 5 × product 各 20、 brand 1 件 ARCHIVED・product 末尾を DISCONTINUED・画像 URL 有無を散らす。

## なぜ必要か ( この ADR の核心 )

### ES では read model は投影。 だから command で作る

read model を直接いじると「投影」という前提が壊れ、 リプレイで消えるしイベント履歴とも食い違う。 command 経由なら、 本番と同じ経路（バリデーション → イベント → projection）を通るので、 **seed データも production の不変条件を必ず満たす**（壊れたデータが入らない）。

### azp 検証が「どのクライアントで認証するか」を縛る

`JwtClientIdValidator` は azp == 期待クライアント ID（`momiji`）のみ通す（[ADR 0002](./0002-エラーの種類と分類方法、gRPC層での取り扱いについて.md) のエラー/認証文脈）。 そのため seeder 専用クライアントを作っても azp が一致せず弾かれる。 現状の単一クライアント前提では **`momiji` クライアントで client_credentials を取るのが唯一素直な道**。

### tool 領域に分離する理由

seed は本番に出すコードではない。 main に置くと app jar に混ざる。 独立ソースセットなら main の生成 stub / 依存を再利用しつつ、 配布物から隔離できる。

## 妥協点 / 検討した代替案

### 代替 A: アプリ内 `@Profile("seed")` の CommandGateway ランナー ( 不採用、 ただし有力 )

CommandGateway を直接叩けば**認証不要**で ES 整合も保てる。 しかし CommandGateway は Axon の全 config 配線が要るため **standalone にできずアプリ内（main）に置く**ことになり、 「tool 領域に分離」要件と相反する。 別プロセスで起動すると projector が二重に走りトークン競合する懸念もある。 「分離」を優先して gRPC 案を採ったが、 認証の一手間が不要な点でアプリ内ランナーは依然有力（要件が変われば再検討）。

### 代替 B: read DB に直接 INSERT ( 却下 )

ES の前提を破壊する（リプレイで消える・イベントと不整合）。

### 代替 C: seeder 専用クライアント / password grant ( 現状不可・不採用 )

専用 service client を作ると azp が一致せず `JwtClientIdValidator` に弾かれる（複数 azp 許可への改修が要る）。 password grant はテストユーザーと directAccessGrants が要る。 どちらも現状の単一クライアント・認可未整備の前提では割に合わない。

## 帰結

良かった点:

- **ES の作法を破らず**テストデータを投入できる（command 経由 → 本番と同じ不変条件を満たす）
- main と分離した tool 領域に隔離（app jar に含まれない）
- 固定 ULID で**冪等・再実行安全**

注意すべき点 ( 重要 ):

- **seeder が BFF の `momiji` クライアントで client_credentials を取る = 実質 BFF クライアントの借用**。 そのクライアントに service account 権限を付与する点も含め、 概念的には smell。 **ローカル限定ツール**として許容している。 専用 service client に分けるには azp 検証の複数許可（= 認可の整備）とセットが必要
- **固定 ULID は共有/ステージング環境を汚染する**。 ローカル投入専用の想定
- `serviceAccountsEnabled` は committed（fresh import 用）。 既存 Keycloak へは partial 反映（`down -v` 不要）か `task reset`
- 実行前提: docker 一式（MySQL / Keycloak / Axon Server）+ backend(`bootRun`) が起動済みであること
