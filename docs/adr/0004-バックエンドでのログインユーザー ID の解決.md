# ADR 0004: バックエンドでのログインユーザー ID の解決（access token の iss + sub → momiji userId）

- **ステータス**: 採用
- **作成日**: 2026-05-31

## コンテキスト

access token を受け取っただけでは「だれがログインしているのか」までは解らない。

backend は API リクエストを受けるたびに「このリクエストは momiji のどのユーザーか」を特定する必要がある。
受け取った **access token（JWT）** から momiji 内部の `userId` を解決する方法を決める。

解決した `userId` は、**対象リソースの取得や認可**（このリクエストは誰のデータを扱うか／操作を許してよいか）に使う。

## 決定

access token から **`iss`（issuer）と `sub`（subject）** を取り出し、その **(iss, sub) の組**で `LOOKUP_EXTERNAL_IDENTITIES` を引いて内部 `userId` を解決する。

1. access token から `iss`（issuer）と `sub`（subject）を取り出す（どちらか欠けていればエラー）。
2. その (iss, sub) の組で `LOOKUP_EXTERNAL_IDENTITIES` を検索する。
3. 検索結果で分岐する。
   - **一致あり** → その `user_id` を返す。
   - **一致なし** → `BusinessException`「ユーザーが登録されていません」を投げる（fail-closed）。

解決した `userId` を起点に、以降のユースケースがユーザー情報を引いたり、認可に使用する。

### access token に `sub` が入るように

残念なことに `sub` claim は access token に必ず入っているとは限らず、IdP によって異なる。

| IdP | access token の `sub` |
|---|---|
| Cognito | ✅ 入っている |
| Auth0 | ✅ 入っている |
| Keycloak | ❌ 入っていない |

そのため、ローカルで使う Keycloak では、`protocolMappers` に `oidc-sub-mapper` を**明示設定**し、access token / ID token / introspection token のすべてに `sub` を乗せている。

> local/keycloak/momiji-realm.json に設定が記述されています。

## 補足

### 1. なぜ sub 単独でなく (iss, sub) の組か

`sub` が一意なのは **同じ IdP の中だけ**。  
IdP をまたぐと、別人どうしで `sub` が衝突しうる。  

そのため (iss, sub) の組で引く。こうしておけば、将来 IdP を移行・併用しても（例: Cognito → Auth0）、`LOOKUP_EXTERNAL_IDENTITIES` のユニーク制約が衝突せず、取り違えも起きない。

### 2. なぜ access token から issuer と sub 属性を取るのか

ユーザー ID 解決（`UserIdResolver`）は、**ユーザーを特定したいユースケースで呼ばれるたび**に走る（全リクエストで必ず走るわけではない）。
この解決はなるべく軽くしたい。
そこで **access token を JWT としてオフライン検証し、その場で iss / sub を読む**方式を採る。IdP への往復が無いぶん速い。

検討したが採らなかった代替は 2 つ。

#### 代替 A: userinfo endpoint に問い合わせる

userinfo は OIDC で `sub` が必須。
そのため access token に `sub` が乗るか（IdP 依存）に左右されず、sub と issuer を確実に取れる。
ただし解決のたびに **IdP へネットワーク往復**が発生し、レイテンシと IdP 負荷が増える。
→ 「IdP 非依存」より「速度」を取り、不採用。

#### 代替 B: access token に momiji の `userId` を直接埋める

トークンを読むだけで解決でき、DB 参照すら要らない。
ただし問題が多い。

- IdP 側で **カスタム claim を作り込む**必要がある
- その `userId` を埋めるには、**IdP のユーザー作成を backend の API 経由で行い**、momiji の `userId` を IdP に同期させる必要がある
- **ソーシャルログインでは対応できない**（ユーザーは外部のソーシャル IdP 側で作られ、momiji の `userId` を claim に注入できない）

→ IdP への密結合とソーシャル非対応が重く、不採用。

## 帰結

### 良かった点：

- access token のオフライン検証だけで `userId` 解決が完結する（**軽量・低レイテンシ・IdP 問い合わせ不要**）
- (iss, sub) の組で引くため、IdP をまたいでも本人を取り違えない（グローバル一意）

### 注意すべき点：

- 「access token に `sub` が必ず乗る」と思い込まない（OIDC が必須にするのは ID Token と userinfo だけ）IdP を変更する際に調査が必要。
