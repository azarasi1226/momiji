# ADR 0002: エラーの種類と分類（ValidationError / BusinessError / UnknownError）

- **ステータス**: 採用
- **作成日**: 2026-05-29

## コンテキスト

エラーは発生する局面が異なる。

- 入力値の検証（ユースケースに入る前。値オブジェクトの validation）
- 業務ルール違反（ユースケースの中。例: ユーザー未存在・メール重複）
- 想定外（上記以外の予期せぬ例外）

これらを場当たり的に個別対応すると、返却方法やクライアントでの扱いがバラバラになり、一貫性が失われる。
かといってすべてを1つのエラー型に統一すると、今度はそのエラー固有の文脈が失われ、扱いが大雑把になりすぎる。
そこで両極の間を取り、**「すべてのエラーは必ず次の3種のどれかに分類される」** という粒度で枠を決める。
細かすぎる個別対応と、粗すぎる完全統一の中間として、3種に集約することで一貫性と文脈を両立させる。

なお、本 ADR はエラーの抽象的な分類を定義しただけのものなので、実装方法や gRPC を用いた返却方法などは解説しない。

## 決定

本システムが扱うエラーは次の3種類に分類する。

| 種類 | いつ | 例 | 対応する例外 |
|---|---|---|---|
| **ValidationError** | ユースケースの**前**に検証する入力値エラー（値オブジェクトの validation） | 名前が空・電話番号の形式不正 | `ValidationException` |
| **BusinessError** | ユースケース**内**で発生する業務エラー | ユーザー未存在・メール重複 | `BusinessException` |
| **UnknownError** | 上記いずれにも該当しない想定外のエラー | NullPointerException・DB 接続断 等 | （上記以外の例外） |

- **ValidationError は ADR 0001 に従いエラーを蓄積する**（fail-fast でなく `List<ValidationError>`）。
  1リクエストで複数の入力エラーをまとめて返せる。
- **BusinessError** はユースケース内のルール違反。原則1件ずつ送出する。
- **UnknownError** は「分類し損ねた残り全部」を受け止める安全網。
  想定外の例外はすべてここに落ちる。

## 帰結

### 良かった点:

- **必ず3種に収まるので、ハンドリングが「脳死」で書ける。**
  新しいエラーを足すときも「3種のどれか」を選ぶだけで、扱いが既定のレールに乗る。
- **「分類し損ねた残り」を UnknownError が受け止める**ので、構造化から漏れるエラーが存在しない。

### 注意すべき点:

- 3種・共通枠に押し込めるため、**エラー単位の個別最適（特定エラーだけ別扱い）はやりにくい。**
  画一性と引き換えの自覚的な妥協。
  gRPC の status code を固定する判断（ADR 0012）も、この妥協の延長線にある。
