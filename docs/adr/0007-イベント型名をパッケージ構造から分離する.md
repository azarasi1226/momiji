# ADR 0007: イベントの論理型名を `@Event` で固定し、パッケージ構造から分離する

- **ステータス**: 採用 ( 全 user イベントに `@Event(namespace = "momiji.user", name = ...)` を付与 )
- **作成日**: 2026-06-03
- **関連 ADR**: [0005 統合テストと単体テスト](./005-バックエンドのテストの考え方。統合テストと単体テストを一緒にした理由), [0006 Cognito](./0006-coginoto)

## コンテキスト

momiji の backend は Axon Framework 5.1.1 を使った event sourcing 構成。 CommandHandler の `@EventSourced` State は USER_ID タグでイベントを再生して `created` を判定する ( 例: [UpdateUserCommandHandler](../../backend/src/main/kotlin/jp/momiji/feature/user/update/UpdateUserCommandHandler.kt) )。 projection も `@EventHandler` でイベントを受けて lookup / read model を更新する。

Axon 5 では各イベントは **論理型名 `QualifiedName` ( namespace + localName )** で識別される。 イベントストアの読み出しは「保存済みイベントの型名」と「ハンドラ引数クラスを `MessageTypeResolver` で前方解決した型名」の一致でマッチングし、 一致したものだけが `@EventSourcingHandler` / `@EventHandler` に配送される。

Axon 5.1.1 ( axon-spring-boot-starter ) の既定 `MessageTypeResolver` は `AnnotationMessageTypeResolver` ( `MessagingConfigurationDefaults.defaultMessageTypeResolver` で登録、 追加 Bean 不要 )。 これは `@Event` アノテーションを見るが、 **未指定時は namespace=パッケージ名 / localName=クラスのシンプル名 にフォールバック**する。

### 経緯 ( なぜ ADR にするか )

リファクタリングでイベントのパッケージを `jp.momiji.events` → `jp.momiji.event` にリネームした。 `@Event` 未付与のイベントは型名がパッケージ名由来だったため、 保存済みイベントの型名が変わってしまった:

- リネーム前作成ユーザー: `jp.momiji.events.user` / `UserCreatedEvent`
- リネーム後作成ユーザー: `jp.momiji.event.user` / `UserCreatedEvent`

結果、 **リネーム前に作成されたユーザー ( 例 `01KSW3VM64MTTB4Z4GY8AYRBWW` ) の `UserCreatedEvent` が現在のクラスにマッチせず再生されない**。 State の `created` が `false` のままになり、 update / email 変更などの CommandHandler が NotUser ビジネスエラーを返す。 リネーム後のユーザーは平気なので「特定ユーザーだけ壊れる」 という気付きにくい症状になった。

単なるパッケージ移動という安全に見えるリファクタが、 イベントストア互換性を黙って壊す。 この罠を将来繰り返さないよう、 **イベントの論理型名をコードの物理構造 ( パッケージ・クラス名 ) から切り離す**方針を明文化する。

## 決定

全ての永続化イベントに `@Event` を明示的に付与し、 `namespace` と `name` を**コードの物理構造に依存しない論理名**として固定する。 namespace は user 集約のイベントとして `momiji.user` に統一する。

`name` は**クラスのシンプル名をそのまま**使う ( = Axon の `name` フォールバックと同じ値 )。

```kotlin
@Event(namespace = "momiji.user", name = "UserCreatedEvent")
data class UserCreatedEvent(
    @EventTag(key = MomijiEventTag.USER_ID)
    val id: String,
    val email: String,
)
```

| イベントクラス | namespace | name |
|---|---|---|
| `UserCreatedEvent` | `momiji.user` | `UserCreatedEvent` |
| `UserUpdatedEvent` | `momiji.user` | `UserUpdatedEvent` |
| `UserDeletedEvent` | `momiji.user` | `UserDeletedEvent` |
| `ExternalIdentityLinkedEvent` | `momiji.user` | `ExternalIdentityLinkedEvent` |
| `EmailChangeRequestedEvent` | `momiji.user` | `EmailChangeRequestedEvent` |
| `EmailChangeConfirmedEvent` | `momiji.user` | `EmailChangeConfirmedEvent` |

`version` は省略し `MessageType.DEFAULT_VERSION` ( `"0.0.1"` ) を使う。

## なぜ必要か ( この ADR の核心 )

### 型名がコードの物理構造に縛られている問題

`@Event` 未付与だと、 イベントの wire 上の identity が「パッケージ名 + クラスのシンプル名」 になる。 つまり以下の **本来は内部実装の都合であるべき操作**が、 すべてイベントストアの後方互換性を破壊する:

- パッケージのリネーム / 移動 ( 今回の事故 )
- クラスのリネーム
- ネストクラスへの移動 / トップレベル化

event sourcing では保存済みイベントは不変の事実であり、 後からスキーマやコード構造を変えても**過去のイベントは読めなければならない**。 型名をコード構造に縛ると、 リファクタの自由度と互換性が両立できない。

`@Event` で論理名を固定すれば、 wire 上の identity ( `momiji.user/UserCreated` ) とコード上のクラス ( どのパッケージのどのクラス名でもよい ) が分離され、 **コード構造のリファクタが安全に行える**。

### namespace を `momiji.user` にした理由

- ドメイン ( bounded context ) を表す論理名であり、 Kotlin のパッケージ階層とは独立
- user 集約に属するイベントであることを型名から読み取れる
- 将来別集約のイベントを追加する場合も `momiji.<集約>` で一貫させられる

### name はクラスのシンプル名と一致させる ( `Event` サフィックスも残す )

`name` の値はクラスのシンプル名 ( `UserCreatedEvent` ) をそのまま使い、 `Event` サフィックスも落とさない。 一見「論理名なのにクラス名と同じで分離になっていないのでは」 と思えるが、 **分離 ( 脱・物理構造依存 ) は値がクラス名と違うことではなく、 型名が明示的にピン留めされていることで達成される**。 値が偶然クラス名と一致していても、 後でクラスやパッケージをリネームしても `@Event` の文字列は変わらないので、 refactor 安全性は一切損なわれない。

その上でクラス名と一致させる方が実務的に優れている:

- **規約が機械的**: 「`name` = クラスのシンプル名をコピペ」 で済み、 考える余地がなく、 レビューでも一目で正否を検証できる
- **フォールバックと一致**: この値は `AnnotationMessageTypeResolver` が `name` 未指定時に使うフォールバック ( シンプルクラス名 ) と同じ。 つまり「デフォルトの name を明示的にロックするだけ」 という素直な意味になり、 新規イベントで付け忘れ→後付けしても `name` 部分はブレない
- `Event` サフィックスが wire 上に残ることで、 イベントストアを見たときに「これはイベントである」 と読めるのもむしろ利点

namespace だけはパッケージ階層と独立した論理ドメイン名 ( `momiji.user` ) にして、 パッケージ移動への耐性を確保する。

## 妥協点 / 検討した代替案

### 代替 A: 現行パッケージ ( `jp.momiji.event.user` ) に型名を固定 ( 不採用 )

リネーム後の保存済みイベントとは一致するので、 現在動いているユーザーは移行なしで動き続けるメリットがあった。 しかし「型名 = 今のパッケージ名」 を固定するのは論理名と物理構造の分離になっておらず、 namespace に Kotlin パッケージ名が残るのも本 ADR の趣旨に反する。 クリーンな論理名 ( `momiji.user` ) を選択した。

### 代替 B: 旧パッケージ ( `jp.momiji.events.user` ) に固定して旧ユーザーを救う ( 不採用 )

`01KSW3...` を救えるが、 リネーム後に作成されたユーザーが今度は壊れる ( 型名が逆に不一致になる )。 どちらにせよ既存データの移行が必要になるため、 救済目的で旧パッケージ名に縛る意味がない。

## 帰結

良かった点:

- イベントの wire identity がコードの物理構造から独立し、 **パッケージ移動・クラスリネームが安全**になった
- namespace / name が論理ドメイン名で表現され、 イベントストアを見たときの可読性も上がる
- 既定 resolver が `AnnotationMessageTypeResolver` なので**追加の Bean 設定は不要**

注意すべき点 ( 重要 ):

- **このアノテーション変更自体が、 既存の保存済みイベントの型名と不一致になる**。 旧 `jp.momiji.events.user.*` だけでなく、 リネーム後の `jp.momiji.event.user.*` も `momiji.user/*` には一致しない。 つまり**移行を伴わずにこの変更だけをデプロイすると、 現在動いているユーザーまで再生不能になる**。 アノテーション変更と既存イベントの型名移行は必ずセットでリリースすること
- イベントストアは Axon Server。 既存イベントの型名移行は SQL 直更新ではできず、 Axon 5.1.1 の変換機構を用いる。 dev 環境はイベントストア破棄で対応可
- 一度 `@Event` の値 ( namespace / name / version ) を決めたら、 **それ自体が後方互換の契約**になる。 以後は安易に変えないこと ( 変える場合は再び移行が必要 )
- イベント以外 ( Command / Query ) はストアに永続化されず、 送受信側が同時にコンパイルされるためリネームしても壊れない。 本 ADR の対象は永続化イベントのみ
