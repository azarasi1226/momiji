# ADR 0010: ハードデリートを廃し、ライフサイクル（archive / discontinue + status）を採用する

- **ステータス**: 採用 ( brand: ACTIVE/ARCHIVED、 product: ACTIVE/DISCONTINUED に移行済み。 `DeleteBrand`/`DeleteProduct` は撤去し `ArchiveBrand`/`DiscontinueProduct` に置換 )
- **作成日**: 2026-06-06
- **関連 ADR**: [0008 プロジェクションの日時は @Timestamp から取得する](./0008-プロジェクションの日時はイベントのTimestampから取得する.md)（read model はリプレイで再構築可能、 という同じ前提に立つ）

## コンテキスト

momiji は CQRS + Event Sourcing（Axon 5.1.1 DCB）。 brand / product 集約を追加した当初は **ハードデリート**で実装していた:

- `BrandDeletedEvent` / `ProductDeletedEvent` を発行し、 projector が read テーブルの行を `deleteFrom` で物理削除
- brand 削除時は「紐づく商品が残っていたら拒否」する linked-products ガードを DCB で持っていた

これを運用・拡張する中で、 ハードデリート前提に複数の歪みが見えた。

### 経緯 ( なぜ ADR にするか )

1. **削除の意味が概念ごとに食い違う。** 商品(Product)はカタログ概念。 商品を「カタログから消す」ことと、 在庫やそれを参照する過去の注文が「消える」ことは別物。 現実の在庫はカタログ削除では消えない。 ハードデリートは「全部まとめて消える」しか表現できず、 この食い違いが気持ち悪さとして表面化した。
2. **削除のカスケード/順序が複雑化する。** brand→product→(将来の在庫) と参照が連なると、 「どの順で消すか」「残っていたら拒否か巻き込み削除か」のガードと順序管理が増殖する。
3. **参照整合・履歴が壊れる。** 商品をハード削除すると、 それを参照する過去の注文や read model が参照切れになる。 実務でも商品は「削除」せず「廃番」にするのが普通。
4. **書き込み側 State が boolean の組合せ地獄になりがち。** `created` / `deleted` の bool を増やすと、 状態が増えるたびに `created && !deleted && ...` の判定が膨らむ。

これらは個別のバグではなく「ハードデリートというモデル選択」に起因する。 方針を明文化する。

## 決定

### 1. ハードデリートを廃し、ライフサイクル状態で表す

| 集約 | 状態 | 旧 delete の置換 |
|---|---|---|
| Brand | `ACTIVE` / `ARCHIVED` | `ArchiveBrand`（`BrandArchivedEvent`） |
| Product | `ACTIVE` / `DISCONTINUED` | `DiscontinueProduct`（`ProductDiscontinuedEvent`） |

「未作成」は状態に**入れない**（集約がまだ存在しないだけ）。 書き込み側 State では `XxxStatus?` の **null = 未作成**で表す。 「在庫切れ」のような**導出値も状態にしない**（Stock の数量から導く別概念）。

### 2. 書き込み側 State は boolean でなく status enum

イベントは「事実＝遷移」（`BrandCreated` / `BrandArchived` 等）、 **status はその畳み込み結果**として State に持つ。 イベントに mutable な status フィールドは持たせない。

```kotlin
enum class ProductStatus { ACTIVE, DISCONTINUED }   // 実在する状態だけ。「未作成」は null

class State(var status: ProductStatus?) {            // null = まだ存在しない
    @EventSourcingHandler fun evolve(e: ProductCreatedEvent)      { status = ProductStatus.ACTIVE }
    @EventSourcingHandler fun evolve(e: ProductDiscontinuedEvent) { status = ProductStatus.DISCONTINUED }
}
// ガード: when (status) { null -> notFound; DISCONTINUED -> ...; ACTIVE -> 続行 }
```

### 3. read model は「行を消さず status を更新」

projector は archive / discontinue で `deleteFrom` せず **`status` カラムを UPDATE**する。 `brands` / `products` に `status` 列を追加。 廃番・アーカイブ済みも行として残り、 クエリで `status` 絞り込みできる。

### 4. 削除ガードの撤去

brand の linked-products ガードは**撤去**した。 アーカイブは商品を孤児化しない（商品はそのまま残り、 新規紐づけだけを止める）ため、 「紐づく商品が残っていたら拒否」する必要がなくなった。

### 5. 集約間ルールは「商品作成にはブランドが ACTIVE」の1つだけ

ハードデリート時代のカスケード/順序ガード群は不要になり、 残る cross-aggregate ルールは **商品作成時にブランドが ACTIVE であること**だけ（DCB の `EventCriteria.either` で product_id × brand_id をロードして判定。 [0007 系の型名固定](./0007-イベント型名をパッケージ構造から分離する.md)と同じく `eventQualifiedName()` で型名を導出）。

## なぜ必要か ( この ADR の核心 )

### 「消さない」のはイベントであって read model の行ではない

Event Sourcing では**履歴の正本はイベントストア**で、 read model は使い捨て・いつでも再構築可能（[ADR 0008](./0008-プロジェクションの日時はイベントのTimestampから取得する.md)）。 だから「廃番でも残す」は **write 側（イベント不変・status 遷移）の話**であって、 read 側の行を残す義務とは別。 read model は用途ごとに刈ってよい（後述の代替参照）。

### 削除の複雑さの正体はライフサイクルの不一致

「`ProductDeleted` で在庫まで消えるのが変」と感じるのは、 カタログと在庫の**ライフサイクルが違う**から。 ハードデリートは 1 つの削除に全部を巻き込むが、 ライフサイクル化すると:

- **Stock は「削除」が無く数量が 0 になるだけ** → 削除の悩みが 1 つ丸ごと消える
- **Product はハード削除でなく廃番** → 在庫を売り切るまで残せ、 過去の注文も無事
- **「ブランド削除→商品削除→在庫削除」の順序・カスケード地獄が消える**（そもそも消さない）

### status enum が boolean より優れる理由

状態が 3 つ以上に増えると bool の組合せは爆発し、 不正状態（`created && deleted && ...`）が表現可能になってしまう。 enum なら不正状態を表現不能にでき、 ガードが `when` で読める。 「未作成」を enum 値にせず null にするのは、 「存在しない」は状態ではなく**集約の不在**だから（ドメインの状態集合を汚さない）。

## 妥協点 / 検討した代替案

### 代替 A: ハードデリート + カスケードガード継続 ( 不採用 )

最小構成では動く。 しかし削除順序とガードの組合せを永遠に自前管理する羽目になり、 参照切れ・履歴破壊のリスクが残る。 ライフサイクル化で根本から消した。

### 代替 B: read model も行を残さず削除 ( 部分的に将来両立 )

「廃番でも残す」を read 側に強制する必要はない。 read model は再構築可能なので、 **店頭用 projection は ACTIVE だけ投影して廃番行を持たない**、 管理用 projection は全部持つ、 という用途別の作り分けが可能。 現状は単一 read model に `status` を持たせ全件返す素朴な実装。 **店頭向けの `findSellable`（ACTIVE のみ）/ ページング/ hot-cold 分離は未着手のフォローアップ**（件数増で必要になる）。

### 代替 C: 状態の永続化に `enum.ordinal()` / MySQL ネイティブ ENUM ( 不採用 )

`ordinal()` は定数の並べ替えで全データの意味がズレる古典的バグ。 MySQL ネイティブ `ENUM` 型は値追加・並べ替えが ALTER 必須で罠もある。 **DB には enum の `name`（文字列）を保存**し、 enum は code 側の正本にした（境界での型の使い分け）。

### 代替 D: ブランド付け替え・状態の逆遷移を許す ( 現状は不採用 )

`UpdateProduct` に brandId は持たせず**ブランド付け替え不可**。 また archive / discontinue は**一方向**で、 unarchive / reactivate は未実装。 誤操作の取り消し手段が無い点は既知の制約（必要になれば別コマンドで追加）。

## 帰結

良かった点:

- **削除のカスケード/順序が消滅**。 残る集約間ルールは「商品作成にはブランド ACTIVE」の 1 つだけ
- **履歴・参照が壊れない**（廃番でも行・イベントが残る）。 read model はリプレイで再構築可能（ADR 0008 と整合）
- 書き込み側 State が status enum で読みやすく、 不正状態を表現不能にできる
- 境界ごとの型の使い分けが一貫: **domain enum（正本）/ wire は proto enum（型安全・番号が契約）/ DB は文字列（再構築可）**

注意すべき点 ( 重要 ):

- **read 側が archived/discontinued も全件返す**。 店頭向けの絞り込み（`findSellable`）・ページング・hot/cold 分離は未着手。 件数が増える前に対応が要る
- **read mapping は不正/空 status に対して防御的にすべき**（`else -> UNSPECIFIED`）。 後付けマイグレーションで空 status の行が残ると `valueOf("")` で一覧 API が落ちる。 現状の `brandStatusToProto` / `productStatusToProto` は strict（`valueOf`）のままで、 防御化はフォローアップ
- **一方向遷移**（unarchive / reactivate なし）。 誤操作の取り消し手段が無い
- 後付けで `status` 列を `NOT NULL` 追加すると既存行が空文字で埋まる。 列追加マイグレーションは backfill とセットにすること
