# ADR 0003: 複数のログイン方式が選択されても、一つのユーザーに紐づける仕組み

- **ステータス**: 採用
- **作成日**: 2026-05-31

## コンテキスト

momiji では、次の2つのログイン方式を提供する。

- **IdP 内部 DB 方式**: Cognito / Keycloak / Auth0 などの IdentityProvider が自前で持つ DB を使うログイン
- **ソーシャル IdP（Identity Federation）**: Google や Facebook など外部のソーシャルアカウントを使うログイン

ここで満たしたい要件は、**同じ人物が複数のログイン方式を使い分けても、momiji の内部 user としては 1 つに集約する**こと。  
実際、ふるさとチョイス・Medium・Qiita など多くのサービスが、複数のログイン方式を 1 つのアカウントに束ねるこの方式を採用している。

### 補足：IdP 各サービスの自動 linking 機能

調査の結果、 主要 IdP は **デフォルトでは自動 account linking しない** ( 公式ドキュメント確認済み ):

| サービス | デフォルト | linking を有効化する方法 |
|---|---|---|
| Cognito | しない | `AdminLinkProviderForUser` API、 または pre sign-up Lambda trigger |
| Keycloak | しない | First Login Flow に `Automatically Link Brokered Account` / `Detect Existing Broker User` authenticator を組み込む |
| Auth0 | しない | Rules / Actions / Account Link Extension で実装 |

このため 3 サービスとも「デフォルトは別 user として扱い、 account linking は 開発者がセキュリティ要件を満たした上で意図的に実装する」 設計になっている。

## 決定

momiji はローカル環境で Keycloak、本番環境で Cognito と IdP を使い分けるため、IdP 側の linking 機能に依存すると環境ごとに別々の実装が必要になり、相性が悪い。そのため、**各 IdP 側の自動 linking 機能は使わず、 momiji 内で独自のリンク戦略を実装する。**

アカウント紐づけの**軸はメールアドレス**とする。アカウント作成時に、すでに同じメールアドレスで登録されていた場合は同一人物として扱い紐づける。
一致しなかった場合は、完全に新規のユーザーとして登録する

具体的には、この方針をログインのたびに [CreateUserCommandHandler](../../backend/src/main/kotlin/jp/momiji/feature/user/create/CreateUserCommandHandler.kt) で次のように判定する。
上から順に評価し、最初に当てはまった分岐で処理を決める。

1. **既知の外部アカウントなら、何もしない（冪等）**: その IdP アカウントが既に momiji ユーザーへ紐づいていれば、新規作成やリンクはしない。
2. **メールが未検証なら、拒否する（fail-fast）**: IdP が「確認済み」と保証しないメールは、その時点で弾く。
3. **⭐同じメールの既存ユーザーがいれば、その人にリンクする**: 確認済みのメールが既存ユーザーと一致すれば、同一人物とみなして外部アカウントを追加で紐づける。
4. **どれにも当てはまらなければ、新規ユーザーを作ってリンクする**: 初めての利用者として momiji に新規ユーザーを作成し、その外部アカウントを紐づける。

### 1. セキュリティをどう担保するか

momiji 独自の 自動 linking 機能において account takeover 攻撃を防ぐため、**2 段階の防御**を入れる:

#### 防御 1: ホワイトリストソーシャル IdP のみ受け入れ

- 受け入れる IdP の種類は [IdentityProvider](../../backend/src/main/kotlin/jp/momiji/domain/idp/IdentityProvider.kt) enum で whitelist 列挙 ( 現在 `LOCAL` / `GOOGLE`)のみ
- これにより:
  - 攻撃者が悪意のある OIDC provider を Cognito / Keycloak 側に追加しても、 momiji の whitelist に無い `providerType` は受け入れ拒否される
  - 「ソーシャル IdP 経由なら何でも信頼する」 という危険な前提を排除し、 **明示的に審査した ソーシャル IdP のみを信頼境界内に入れる**  今のところ email、 email_verified 2つの属性を返すソーシャル IdP は Google しか確認できていない

#### 防御 2: ソーシャル IdP が返却する email_verified 属性が true のみ受け入れ

- [CreateUserCommandHandler:32-34](../../backend/src/main/kotlin/jp/momiji/feature/user/create/CreateUserCommandHandler.kt) で `email_verified=false` の sign-in は拒否している
- これにより:
  - 攻撃者が「verified ではない任意 email」 を主張しても拒否される
  - whitelist IdP ( Google ) は信頼境界内なので、 そこから「verified」 と返ってきた email の主張は信頼できる

### 2. ユーザー作成時のログイン種別（IdP 内部アカウント / ソーシャルアカウント）の判定方法

ユーザー作成時に「どのログイン方法が使われたか」を示す属性（`identity_provider` 相当）は、OIDC 仕様上 access token と ID token のどちらにもデフォルトで含まれない。
そのため backend は、この属性を **IdP へ直接問い合わせて取得する**。

| IdP | 問い合わせ API | 取得するもの |
|---|---|---|
| Keycloak | `GET /admin/realms/{realm}/users/{id}/federated-identity` | `[{identityProvider: "google", ...}]` |
| Cognito | `AdminGetUser` の `identities` ユーザー属性 | `[{providerType: "Google", ...}]` |

これらの API から返る結果は配列形式なので、先頭要素 `[0]` を決め打ちで取り出す。

> 配列になっているのは、IdP 側の自動 linking で複数のソーシャル IdP を 1 ユーザーに link した場合に、複数要素が並ぶため。
> momiji は自動 linking を使わない（前述）ため、先頭の決め打ちで問題ない。

## 補足

### 1. なぜ IdP 側の auto linking 機能を使わないか

- 各 IdP の auto linking 設定 ( Cognito の `AdminLinkProviderForUser`、 Keycloak の `Automatically Link Brokered Account` 等 ) は IdP ベンダ依存で、 セキュリティ要件 ( whitelist + email_verified の AND ) を単一のコード上で表現しにくい
- IdP を切り替える ( 例: 将来 Auth0 へ移行 ) たびに linking 戦略を再設計するコストを避けたいから

### 2. なぜメールアドレスベースの自動紐づけを採用したのか

- 「同じ email = 同じ人」 という暗黙の前提は危険そうであるが、「**whitelist IdP が verified と保証した email**」 に限れば許容できると判断
- ユーザー体験上、 「同じ Google アカウントなら同じ momiji ユーザー」 が直感的であるため

### 3. 将来 IdP を追加する時の作業

新しい IdP ( 例: Microsoft、 Facebook ) を whitelist に追加したい場合:

1. `IdentityProvider` enum に 1 行追加
2. `IdentityProviderResolver` の `when` に分岐追加 + 対応 abstract property を 1 個追加 ( **コンパイルエラーで強制される** )
3. 各 IdP Client ( Keycloak / Cognito ) で新 abstract property を override ( **コンパイルエラーで強制される** )

IdentityProvider enum に追加さえすればコンパイルエラーで未実装領域がエラーとなり、修正漏れが無い。

## 帰結

### 良かった点:

- 明らかにユーザーの体験は向上する
- 各 IdP の auto linking 設定差異に依存せず、 一貫した 自動紐づけ 戦略を維持できる
- whitelist + email_verified の 2 ガードが明示的に code 上に現れ、 セキュリティ判断の根拠が追跡可能
- 新 IdP 追加時の作業がコンパイルエラーで強制され、 mapping 漏れの セキュリティホール を作れない

### 注意すべき点:

- email_verified の信頼性は IdP 採用時のレビューに依存。 新 IdP を whitelist に追加する時は「verified email の信頼性」 を必ず審査する
- 将来 自動紐づけ機能を を IdP 側に移すケース ( 例: Cognito の `AdminLinkProviderForUser` を使う運用に変更 ) では `CognitoUserClient` の「`identities` 配列 1 要素前提」 が崩れるため IdP 側の機能は利用今後できないものとする
