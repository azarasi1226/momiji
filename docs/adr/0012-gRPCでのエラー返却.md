# ADR 0012: gRPC でのエラー返却（ErrorDetail / Status.details）

- **ステータス**: 採用
- **作成日**: 2026-05-29
- **関連 ADR**: 0002（エラーの3分類。本 ADR はその3分類を gRPC で返す方法）

## コンテキスト

ADR 0002 で定めた3分類（ValidationError / BusinessError / UnknownError）を、gRPC でクライアントへどう返すかを決める。

gRPC のエラーを `Status.INVALID_ARGUMENT.withDescription(ex.message)` のように文字列だけで返すと、いくつもの不都合がある。

- backend で `ValidationException(errors: List<ValidationError>)` を投げても、gRPC は message を `"[name] 名前は必須です / [phoneNumber] 電話番号は..."` のような join 文字列にしてしまう。
- frontend はエラー文字列を表示するだけで、「どの field のエラーか」を識別できない。
- 結果として、form の field 別ハイライト等の UX 改善ができない。
- 想定外例外も spring-grpc 既定の `UNKNOWN` 扱いになり、クライアントは原因究明できない。

構造化して扱う方針自体は最初から固まっていたため、ここでは **実装コストと運用のバランスが取れたエラー処理機構を取り入れること** を狙いとした。

## 決定

### 1. ErrorDetail を Status.details に載せる

`grpc/momiji/common/v1/error.proto` に `ErrorDetail` を定義し、`oneof` で3分類を表現する。

```proto
// oneof で、3種類のエラーの中から必ず一つだけが返ってくる
message ErrorDetail {
  oneof error {
    ValidationError validation_error = 1;
    BusinessError business_error = 2;
    UnknownError unknown_error = 3;
  }
}
```

backend では `GrpcConfig.grpcExceptionHandler` が送出された例外の型を判定し、対応する `ErrorDetail` に変換して `google.rpc.Status.details` に乗せて返す。

> gRPC のエラーは本来「StatusCode + Message」の2つしか持たない。
> 構造化したエラーを返すための公式な拡張枠が `google.rpc.Status.details` であり、ここに `ErrorDetail` を載せている。

```kotlin
fun grpcExceptionHandler(): GrpcExceptionHandler =
    GrpcExceptionHandler { ex ->
        when (ex) {
            is ValidationException ->
                buildStatusException(Status.INVALID_ARGUMENT, "ValidationError", ex.toErrorDetail())
            is BusinessException ->
                buildStatusException(Status.INVALID_ARGUMENT, "BusinessError", ex.toErrorDetail())
            else -> {
                val correlationId = UUID.randomUUID().toString()
                logger.error(ex) { "予期せぬエラー correlationId=$correlationId" }
                buildStatusException(Status.UNKNOWN, "UnknownError", unknownErrorDetail(correlationId))
            }
        }
    }
```

### 2. status code はほぼ固定でよい

BusinessError と ValidationError はどちらも `INVALID_ARGUMENT`（code `3`）に、UnknownError は `UNKNOWN`（code `2`）にマッピングする。
そのうえで **種類ごとに status code を細かく振り分けることはしない**。
種類の識別は code ではなく `ErrorDetail` 内の構造化されたエラーが担うため、gRPC のエラー番号はほぼ固定で構わない。

## 妥協点と運用設計

### UnknownError の安全な出し方（重要）

想定外例外の `ex.message` をそのままクライアントに返すと、内部情報がブラウザの DevTools 経由で漏洩する。

| 想定外例外の例 | `.message` の中身 | 漏れる情報 |
|---|---|---|
| `JdbcSQLException` | `Column 'password_hash' not found` | 内部スキーマ |
| `RestClientException` | `Connection refused: cognito.internal.amazon.com:443` | 内部ホスト名 |
| `FileNotFoundException` | `/var/secrets/db.key not found` | ファイルパス |
| `NullPointerException` | `Cannot invoke User.email() because user is null` | コード構造 |

このため `UnknownError` は **固定メッセージ + correlationId（UUID）** だけを返し、詳細はサーバーログにのみ出す。

```proto
message UnknownError {
  string message = 1;        // 固定 "サーバーエラーが発生しました"
  string correlation_id = 2; // backend ログとの突合用 UUID
}
```

#### 運用フロー

1. **[クライアント]** 「サーバーエラーが発生しました（問い合わせ番号: {correlationId}）」を受け取る
2. **[クライアント]** サポートに問い合わせ番号を伝える
3. **[サポート]** backend ログを `grep {correlationId}` で検索
4. **[開発チーム]** `correlationId` で検索をかけ原因箇所を特定

ユーザーには内部情報を見せず、サポート診断性は維持される。
これが妥協できるラインと判断した。

## 帰結

### 良かった点

- **`oneof` + `else → UnknownError` で、構造化から漏れる例外が存在しない。**
  proto の `oneof` で「3種のうち必ず1つ」が保証され、ハンドラの `else` 分岐が想定外例外を漏れなく `UnknownError` へ落とす。
  「分類し忘れて素の string が返る」経路が型と分岐の両方で塞がれる。
- **固定メッセージ + correlationId で、情報漏洩なしにサポート診断が可能。**
  内部スキーマ・ホスト名・パス等を出さずに、ログ突合での原因特定は維持される。
- frontend は `lib/grpc-error.ts` で `ErrorDetail` を復元し、field 別ハイライト等に使える。

### 注意すべき点

- **status code を固定したため、エラー単位の拡張性は下がっている。**
  「特定のエラーだけ別の gRPC code を返したい」「このエラーにだけ追加メタデータを持たせたい」といった個別最適は、共通の枠組みと衝突する。
  画一性と引き換えに、はみ出すケースへの柔軟性を捨てている、という自覚的な妥協。
