# ADR 0002: エラーの種類と分類方法、gRPC層での取り扱いについて

- **ステータス**: 採用
- **作成日**: 2026-05-29

## コンテキスト

gRPC のエラーを `Status.INVALID_ARGUMENT.withDescription(ex.message)` のように文字列で返すだけだと、 いくつもの不都合がある:

- backend で `ValidationException(errors: List<ValidationError>)` を投げても、 gRPC は message を `"[name] 名前は必須です / [phoneNumber] 電話番号は..."` のような join 文字列にしてしまう
- frontend は string を表示するだけで、 「どの field のエラーか」 を識別できない
- 結果として、 form の field 別ハイライト等の UX 改善ができない
- 想定外例外も spring-grpc 既定の `UNKNOWN` 扱いになり、 クライアントは原因究明できない

エラーを構造化して扱うという方針自体は最初から固まっていたため、 ここでは **実装コストと運用のバランスが取れたエラー処理機構を取り入れること** を狙いとした。

## 決定

### 1. エラーの種類

本システムが扱うエラーは次の 3 種類に分類する。

| 種類 | 分類方法 |
|---|---|
| ValidationError | ユースケースの**前**に検証する入力値エラー (値オブジェクトの validation) |
| BusinessError | ユースケース**内**で発生しうる業務エラー (例: ユーザー未存在、 メール重複) |
| UnknownError | 上記いずれにも該当しない、 原因不明のエラー (想定外の例外) |

### 2. gRPC通信上でのエラーの返却方法

`grpc/momiji/common/v1/error.proto` に `ErrorDetail` メッセージを定義し、 `oneof` で各エラー型を表現している。  

```proto
// oneof で、3種類のエラーの中から必ず一つだけが返ってくる
message ErrorDetail {
  oneof error {
    ValidationError validation_error = 1;
    BusinessError business_error = 2;
    UnknownError unknown_error = 3;
  }
}
```

backend では `GrpcConfig.grpcExceptionHandler` が送出された例外の型を判定し、 対応する `ErrorDetail` に変換して `google.rpc.Status.details` に乗せて返す。

> gRPC のエラーは本来 「StatusCode + Message」 の 2 つしか持たない。 構造化したエラーを返すための公式な拡張枠が `google.rpc.Status.details` であり、 ここに上記の `ErrorDetail` を載せている。

```kotlin
  fun grpcExceptionHandler(): GrpcExceptionHandler =
        GrpcExceptionHandler { ex ->
            when (ex) {
                is ValidationException ->
                    buildStatusException(
                        Status.INVALID_ARGUMENT,
                        "ValidationError",
                        ex.toErrorDetail(),
                    )
                is BusinessException ->
                    buildStatusException(
                        Status.INVALID_ARGUMENT,
                        "BusinessError",
                        ex.toErrorDetail(),
                    )
                else -> {
                    val correlationId = UUID.randomUUID().toString()
                    logger.error(ex) { "予期せぬエラー correlationId=$correlationId" }
                    buildStatusException(
                        Status.UNKNOWN,
                        "UnknownError",
                        unknownErrorDetail(correlationId),
                    )
                }
            }
        }
```

## 妥協点と運用設計

### 1. `UnknownError` の安全な出し方 (重要)

想定外例外の `ex.message` をそのままクライアントに返すと、 内部情報がブラウザの DevTools 経由で漏洩する:

| 想定外例外の例 | `.message` の中身 | 漏れる情報 |
|---|---|---|
| `JdbcSQLException` | `Column 'password_hash' not found` | 内部スキーマ |
| `RestClientException` | `Connection refused: cognito.internal.amazon.com:443` | 内部ホスト名 |
| `FileNotFoundException` | `/var/secrets/db.key not found` | ファイルパス |
| `NullPointerException` | `Cannot invoke User.email() because user is null` | コード構造 |

このため `UnknownError` は **固定メッセージ + correlationId (UUID)** だけを返し、 詳細はサーバーログにのみ出す:

```kotlin
val correlationId = UUID.randomUUID().toString()
logger.error(ex) { "予期せぬエラー correlationId=$correlationId" }
buildStatusException(
    Status.UNKNOWN,
    "サーバーエラーが発生しました",
    buildUnknownDetail(correlationId),
)
```

```proto
message UnknownError {
  string message = 1;       // 固定 "サーバーエラーが発生しました"
  string correlation_id = 2; // backend ログとの突合用 UUID
}
```

#### 運用フロー

1. **[クライアント]** ：「サーバーエラーが発生しました (問い合わせ番号: {ErrorCode})」 を 受け取る
2. **[クライアント]** ：サポートに問い合わせ番号を伝える
3. **[サポート]** ： backend ログを `grep 7f3a8e2c` で検索
4. **[開発チーム]** : "`correlationId={ErrorCode}` で検索をかけ原因箇所を特定

ユーザーには内部情報を見せず、 サポート診断性は維持される。 これが 妥協できるラインと判断

### 2. gRPC のエラー番号はほぼ固定されるが、 それで良い

BusinessError と ValidationError はどちらも gRPC Status `INVALID_ARGUMENT` (code `3`)、 UnknownError は `UNKNOWN` (code `2`) に固定的にマッピングされ、 **エラーの種類ごとに gRPC ステータスコードを細かく振り分けることはしない**。
種類の識別は code ではなく `ErrorDetail` 内の構造化されたエラーが担うため、 gRPC のエラー番号はほぼ固定で構わない。

## 帰結

### 良かった点

- **エラーが必ず3種に収まるので、ハンドリングが「脳死」で書ける**  
  新しいエラーを足すときも「ValidationError / BusinessError / UnknownError のどれか」を選ぶだけで、
  返却方法・gRPC ステータス・frontend での扱いはすべて既定のレールに乗る。
  分類さえ決まれば実装が機械的に決まり、毎回「このエラーはどう返すべきか」を考えなくて済む。

- **`oneof` + `else → UnknownError` で、構造化から漏れる例外が存在しない**  
  proto の `oneof` で「3種のうち必ず1つ」が保証され、ハンドラの `else` 分岐が
  想定外例外を漏れなく `UnknownError` へ落とす。
  「分類し忘れて素の string が返る」経路が型と分岐の両方で塞がれているため、
  クライアントは常に構造化されたエラーを受け取れる。

- **`UnknownError` の固定メッセージ + correlationId 構造で、情報漏洩なしでサポート診断が可能**  
  （内部スキーマ・ホスト名・パス等を出さずに、ログ突合での原因特定は維持される）

### 注意すべき点

- **共通化と引き換えに、エラー単位の拡張性は下がっている**  
  全エラーを3種・固定ステータスコード・共通 `ErrorDetail` に押し込めたため、
  「特定のエラーだけ別の gRPC code を返したい」「このエラーにだけ追加メタデータを持たせたい」
  といった*個別最適*をやろうとすると、共通の枠組みと衝突する。
  画一性（脳死で書ける）と引き換えに、はみ出すケースへの柔軟性を捨てている、という自覚的な妥協。
